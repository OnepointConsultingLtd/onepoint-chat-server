# Conversation ID Management Analysis

## Overview
This document explains how `conversationId` is generated, stored, and managed throughout the application, including the relationship between localStorage `sessionId` and database storage.

## Key Components

### 1. **LocalStorage Storage** (`onepoint-chat-ui/src/lib/persistence.ts`)

The `conversationId` is stored in localStorage under the key `SESSION_ID`:

```typescript
// Storage key
PersistenceKeys.SESSION_ID = 'sessionId'

// Save conversationId to localStorage
saveConversationId(conversationId: string | null) {
  localStorage.setItem(PersistenceKeys.SESSION_ID, conversationId);
}

// Retrieve conversationId from localStorage
getConversationId(): string | null {
  return localStorage.getItem(PersistenceKeys.SESSION_ID);
}
```

**Important**: The localStorage key is `'sessionId'`, but it stores the `conversationId`.

---

### 2. **Conversation ID Generation** (Server-side)

The conversationId is generated by the `@gilf/chat-websocket-server` library when:
- A client sends a `'request-conversation-id'` message
- The WebSocket server creates a new conversation and generates a UUID

**Location**: External library (`@gilf/chat-websocket-server`)
- The library handles WebSocket connections and conversation management
- It generates UUIDs for new conversations
- The ID is sent back to the client via `'conversation-id'` message

---

### 3. **Client-Side Flow** (`onepoint-chat-ui/src/hooks/useChat.ts`)

#### Initial Request Flow:
```typescript
// When user sends first message without conversationId
sendMessageToServer(text: string) {
  if (!currentConversationId.current) {
    messageQueue.current.push({ text: text.trim() });
    sendMessage(wsRef.current, 'request-conversation-id', '', '');
    return; // Wait for conversationId
  }
  // ... send message with conversationId
}
```

#### Receiving Conversation ID:
```typescript
case 'conversation-id': {
  const { conversationId } = message;
  const lastConversationId = getConversationId(); // From localStorage
  
  if (lastConversationId !== conversationId) {
    currentConversationId.current = conversationId;
    saveConversationId(conversationId); // Save to localStorage as 'sessionId'
    
    // If there was a previous conversationId, load its history
    if (conversationId && lastConversationId) {
      loadChatHistory(lastConversationId)
        .then((serverMessages) => {
          // Import old history into new conversation
          sendMessage(wsRef.current, 'import-history', 
            { history: serverMessages }, conversationId);
        });
    }
  }
}
```

#### Saving Chat History:

```typescript
  const newConversationId = conversationId;
  
  // Check if conversation exists
  const existingConversation = await collection.findOne({ 
    conversationId: newConversationId 
  });
  
  if (existingConversation) {
    // UPDATE: Merge new messages with existing chat history
    await collection.updateOne(
      { conversationId: newConversationId },
      { $set: { chatHistory: mergedChatHistory, ... } }
    );
  } else {
    // CREATE: New conversation document
    await collection.updateOne(
      { conversationId: newConversationId },
      { $set: { conversationId: newConversationId, chatHistory: ... } },
      { upsert: true }
    );
  }
  
  // DELETE old conversation if different
  if (oldConversationId && oldConversationId !== newConversationId) {
    await collection.deleteOne({ conversationId: oldConversationId });
  }
}

```

```
│                    CLIENT SIDE                              │

1. Receive 'request-conversation-id'
   ↓
2. Generate new UUID (via @gilf/chat-websocket-server)
   ↓
3. Send 'conversation-id' message back to client
   ↓
4. Receive messages with conversationId
   ↓
5. Process messages through callbacks
   ↓
6. Save to MongoDB via saveConversationHistory callback
   ├─→ Uses conversation.id as conversationId
   └─→ Calls saveChatHistory(chatHistory, conversationId, ...)
```

## Critical Issues & Behavior

### Issue 1: ConversationId Changes on Refresh

**Problem**: The conversationId can change on every page refresh.

**Root Cause**:
1. On page refresh, `currentConversationId.current` is initialized as `null` (line 33 in `useChat.ts`)
2. **CRITICAL**: The conversationId from localStorage is **NEVER** restored into `currentConversationId.current` on component mount
3. When user sends a message, `currentConversationId.current` is `null`, so it sends `'request-conversation-id'`
4. WebSocket server generates a **NEW** conversationId (UUID)
5. Client receives new ID and saves it to localStorage, **overwriting** the old one
6. The old conversation in the database gets deleted (line 157-160 in `handleApi.ts`)

**Code Evidence**:
```typescript
// useChat.ts line 33 - Always starts as null
const currentConversationId = useRef<string | null>(null);

// useChat.ts line 272 - No restoration from localStorage
if (!currentConversationId.current) {
  messageQueue.current.push({ text: text.trim() });
  sendMessage(wsRef.current, 'request-conversation-id', '', ''); // Always requests new ID!
  return;
}
```

**Missing Code** (should exist but doesn't):
```typescript
// This should happen on component mount but doesn't:
useEffect(() => {
  const savedId = getConversationId(); // Get from localStorage
  if (savedId) {
    currentConversationId.current = savedId; // Restore it!
  }
}, []);
```

**Code Evidence**:
```typescript
// In useChat.ts - when receiving new conversationId
if (lastConversationId !== conversationId) {
  // New ID received - old one is different
  currentConversationId.current = conversationId;
  saveConversationId(conversationId); // Overwrites localStorage
  
  // Old conversation gets deleted in saveChatHistory:
  if (oldConversationId && oldConversationId !== newConversationId) {
    await collection.deleteOne({ conversationId: oldConversationId });
  }
}
```

### Issue 2: SessionId vs ConversationId Confusion

**Problem**: The localStorage key is `'sessionId'` but it stores `conversationId`.

**Impact**: 
- Naming confusion
- The term "sessionId" suggests it's tied to a user session, but it's actually a conversation identifier
- This can lead to bugs if developers expect session-based behavior

### Issue 3: No User Association

**Problem**: Conversations are not associated with authenticated users.

**Current State**:
- Conversations are stored by `conversationId` only
- No `userId` or `sessionId` field in the database
- Multiple users can access the same conversation if they have the ID
- No way to list a user's conversations

**Evidence**:
```typescript

  chatHistory: [...],
  userMessage: "...",
  timestamp: "..."
```

## How Conversations Are Saved

### Step-by-Step Process:

1. **Message Sent** (`useChat.ts`):
   ```typescript
   sendMessage(wsRef.current, 'message', text, currentConversationId.current);
   ```

2. **Server Processes** (`src/index.ts`):
   - WebSocket receives message with conversationId
   - Processes through `onepointCallback` (adds system prompts, context)
   - Returns chatHistory with referenceSources

3. **Save Callback Triggered** (`src/callbacks/saveConversationHistory.ts`):
   ```typescript
   saveConversation(conversation: Conversation, referenceSources?: any[]) {
     const conversationId = conversation.id; // From WebSocket server
     await saveChatHistory(chatHistory, conversationId, referenceSources);
   }
   ```

4. **Database Save** (`src/api/handleApi.ts`):
   - Checks if conversation exists by `conversationId`
   - Updates existing or creates new document
   - Deletes old conversation if ID changed

## Recommendations

### 1. **Persist ConversationId Across Refreshes**
- Ensure the WebSocket server recognizes existing conversationIds
- Don't generate new IDs if a valid one exists in localStorage
- Consider sending the existing conversationId on WebSocket connection

### 2. **Add User Association**
- Store `userId` in database when user is authenticated
- Link conversations to user accounts
- Allow users to see their conversation history

### 3. **Rename localStorage Key**
- Change `SESSION_ID` to `CONVERSATION_ID` for clarity
- Update all references throughout the codebase

### 4. **Prevent Accidental Deletion**
- Don't delete old conversations automatically
- Instead, mark them as "archived" or "inactive"
- Allow manual cleanup or expiration policies

### 5. **Add Conversation Restoration**
- On page load, check localStorage for conversationId
- Attempt to restore conversation from database
- Only create new conversation if restoration fails

## Summary

The `conversationId` flow:
1. **Generated** by `@gilf/chat-websocket-server` as a UUID
2. **Stored** in localStorage under key `'sessionId'`
3. **Used** in all WebSocket messages and database operations
4. **Saved** to MongoDB with the conversation document
5. **Can change** on refresh if the server generates a new one
6. **Old conversations deleted** when a new ID is assigned

The main issue is that the conversationId is not properly persisted across page refreshes, leading to conversation loss and ID changes.
